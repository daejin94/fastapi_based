# Workflow.md

## 1. 기본 원칙

- 모든 대화와 문서 작성은 한글로 진행한다.
- 코드 작성 전 반드시 현재 상태를 먼저 파악한다.
- 코드 작성 전 사용자에게 현재 상태와 예정 작업을 요약 보고한다.
- 사용자의 승인 전에는 실제 파일 수정 및 코드 작성을 시작하지 않는다.
- 기존 프로젝트 구조와 관례를 최우선으로 따른다.
- 문서와 코드의 싱크를 항상 유지한다.

---

## 2. 작업 시작 전 필수 문서 확인

작업 시작 전에 다음 문서를 먼저 확인한다.

### 공통 문서

- docs/spec.md
- docs/plan.md
- docs/progress.md
- docs/next.md

위 문서를 통해 다음을 파악한다.

- 전체 요구사항
- 현재 구현 상태
- 진행 중인 작업
- 다음 우선 작업

---

## 3. 작업 유형별 추가 문서 확인

### 기능 구현 시

- docs/feature/ 폴더 내 관련 문서를 읽고 상세 요구사항을 확인한다.

### 리팩토링 시

- docs/refactoring/ 폴더 내 관련 문서를 읽고 개선 목적과 범위를 확인한다.

### 에러 대응 시

- docs/troubleshooting.md 를 먼저 확인하여 유사 문제가 있었는지 확인한다.

---

## 4. 코드 작성 전 상태 보고

코드를 작성하기 전에 반드시 현재 상태를 정리하여 사용자에게 보고한다.

### 보고 내용

- 현재 요구사항 요약
- 확인한 문서 목록
- 현재 코드 상태
- 이번 작업에서 수정할 범위
- 예상 변경 파일
- 문서 업데이트 대상
- 다음 작업 계획

보고 후 반드시 아래 질문을 한다.

코드 작성 진행할까?

사용자가 승인하면 그때 실제 코드 수정을 시작한다.

---

## 5. 승인 기반 실행 규칙

다음과 같은 답변이 있을 때만 실제 코드 수정 및 파일 생성을 시작한다.

- 네
- 진행해
- ㄱㄱ
- 해줘
- 오케이
- 승인

다음 작업은 승인 없이 가능하다.

- 단순 설명
- 문서 초안 작성
- 폴더 구조 제안
- 코드 예시 제공

단, 실제 저장소 파일을 수정하는 작업은 반드시 승인 후 진행한다.

---

## 6. 작업 중 문서 동기화 규칙

코드와 문서가 어긋나지 않도록 작업 중에도 문서를 갱신한다.

### 수시 업데이트 대상

#### docs/plan.md

- 구현 단계 변경
- 설계 변경
- 작업 순서 변경

#### docs/feature/

- 기능 구현 상세 내용 변경 시

#### docs/refactoring/

- 리팩토링 범위 또는 목적 변경 시

#### docs/troubleshooting.md

- 에러 해결 시 원인과 해결 방법 기록

---

## 7. 작업 완료 후 문서 정리

작업이 끝나면 다음 문서를 반드시 정리한다.

### docs/progress.md

- 완료된 항목을 Done으로 이동
- 진행률 업데이트

### docs/next.md

- 다음 작업을 구체적으로 기록

### docs/decisions.md

- 기술적 선택 이유 기록
- 왜 그렇게 구현했는지 설명

### docs/troubleshooting.md

- 발생한 오류
- 원인
- 해결 방법
- 처리 결과 기록

---

## 8. 출력 방식 규칙

작업 성격에 따라 출력 방식을 다르게 한다.

### 단순 작업

예

- 폴더 생성
- 파일 이동
- 설정 추가
- 반복 코드 생성

이 경우

- 설명을 최소화한다
- 결과와 필요한 명령어만 제공한다

### 설계 판단이 필요한 작업

예

- 인증 구조 설계
- 데이터 모델 설계
- 상태 관리 방식 변경
- 아키텍처 변경

이 경우

- 판단 근거를 간단히 설명한다
- 핵심 이유만 짧게 정리한다

---

## 9. 명령어 제공 규칙

모든 명령어는 터미널에서 바로 실행할 수 있어야 한다.

### 규칙

- 명령어는 항상 독립된 코드 블록으로 제공한다.
- 여러 명령어를 설명과 섞어 작성하지 않는다.

---

## 10. FastAPI 프로젝트 구조 원칙

FastAPI 프로젝트는 다음 구조를 우선 사용한다.

- app/main.py
- app/api/
- app/services/
- app/repositories/
- app/models/
- app/schemas/
- app/core/
- app/db/
- app/dependencies/
- app/utils/
- tests/

새 파일은 반드시 기존 프로젝트 구조 안에 생성한다.

이미 비슷한 역할의 파일이 있다면 새로 만들지 말고 확장한다.

---

## 11. 아키텍처 규칙

기본 흐름

Router → Service → Repository → Database

### Router 역할

- HTTP 요청 처리
- 입력 검증
- Service 호출
- 응답 반환

### Service 역할

- 비즈니스 로직 처리
- 도메인 검증
- Repository 조합

### Repository 역할

- 데이터베이스 접근
- ORM 쿼리 처리

---

## 12. 금지 사항

다음 작업은 허용되지 않는다.

- Router에서 DB 직접 접근
- Router에서 비즈니스 로직 작성
- ORM 모델을 API 응답으로 직접 반환
- 기존 구조와 다른 새로운 레이어 생성
- 동일 목적의 유틸 파일 중복 생성
